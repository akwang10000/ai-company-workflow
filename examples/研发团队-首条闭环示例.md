# examples/研发团队-首条闭环示例.md

## 目标
提供一条从任务输入到交付收口的首条闭环样板，帮助人和 Codex 理解整套研发团队版流程怎么串起来。

---

## 这份文档解决什么问题
它不是单个角色手册，也不是单个 workflow 说明，而是把：
- 任务单
- 角色分工
- workflow 流转
- 交接说明
- 审核与收口

串成一条可参考执行的完整样板。

## 首条闭环样板（feature）
### 场景
新增一个后台用户封禁记录查询页。

### 流转顺序
1. CEO / 用户提出业务目标
2. PM Agent 澄清需求，补齐验收标准
3. Tech Lead Agent 输出实现方案与风险说明
4. Dev Agent 完成功能实现并附自测结果
5. QA / Review Agent 对照验收标准审核
6. CEO / 用户确认是否接受当前交付范围
7. Ops / Release Agent 汇总交付摘要并归档

---

## 每一步最少要产出什么
### 1. CEO / 用户
- 任务标题
- 目标
- 优先级

### 2. PM Agent
- 范围 / 非范围
- 验收标准
- handoff_note 给 Tech Lead

### 3. Tech Lead Agent
- 技术方案摘要
- 风险点
- handoff_note 给 Dev

### 4. Dev Agent
- 实现结果
- 修改点说明
- 自测结果
- handoff_note 给 QA / Review

### 5. QA / Review Agent
- 审核结论
- 返工意见或通过说明
- handoff_note 给 CEO / 用户

### 6. CEO / 用户
- 是否接受当前交付范围
- 是否要求继续修改

### 7. Ops / Release Agent
- final_result
- artifacts
- 交付摘要
- 归档记录

---

## 这条样板的意义
它的价值不在“举了一个例子”，而在于证明：
- 任务不是一句话扔给 Dev 就结束
- 每一步都要有负责人
- 每一次流转都要有说明
- 每个阶段都要有可检查产物
- 最终能形成一条完整闭环

---

## 当前缺口
虽然这条闭环样板已经能跑通 feature 主链路，但为了更接近“可交给 Codex 顺序执行”的第一版，后续还应继续补：
- 缺陷优先级 / 事故分级规则
- 回归验证模板
- Ready for Delivery 检查清单
