# AGENTS.md

## 目的
本文件定义本项目中智能体 / Codex 的**项目级执行硬规则**。

它不是项目介绍，也不是角色手册或 workflow 说明书。
它只回答四件事：
- 什么情况下可以执行
- 执行时必须遵守什么
- 什么情况下必须停下
- 什么样的产出才算有效推进

---

## 适用范围
本文件对本项目所有任务都生效，包括但不限于：
- feature
- bugfix
- hotfix
- research
- 后续扩展场景下的其他可流程化任务

若局部文档与本文件冲突：
- **安全 / 审批 / 停止规则，以本文件为准**
- 具体任务内容、角色动作、流程步骤，再参考对应文档

---

## 项目当前阶段约束

### 1. 第一阶段只先跑通研发团队版
当前优先目标不是覆盖全部行业，而是先把研发团队版跑通。

### 2. 先闭环，再扩展
优先保证：
- 任务能从输入走到交付
- 每个环节有负责人
- 每次流转有状态变化
- 每次交接有明确说明

不要为了“更聪明”牺牲可执行性。

### 3. 文档必须服务执行
本项目所有文档，优先服务：
- 可执行
- 可检查
- 可交接
- 可验收

不能只追求“写得像文档”，而要追求“代理能照着做”。

### 4. 编码格式统一为 UTF-8
本项目中的所有文档、代码、脚本统一使用 **UTF-8** 编码。

在创建、编辑、批量改写、迁移文件时：
- 必须显式保持 UTF-8
- 不得混用本地默认编码、ANSI、GBK 等隐式写回方式
- 涉及批量处理时，先小范围验证，再全量执行

若编码处理存在不确定性：
- 先暂停批量修改
- 先抽样读回确认
- 不得为了省事牺牲文档可读性

## 执行入口规则

### 1. 没有任务单，不进入正式执行
若缺少任务单或任务草稿，不得假装已经进入执行阶段。

### 2. 至少具备以下字段，才可视为可执行任务
- `title`
- `goal`
- `status`
- `current_owner`
- `acceptance_criteria`（或等价完成标准）

若缺失以上关键字段：
- 先补任务单
- 不进入开发 / 修复 / 发布阶段

### 3. 没有验收标准，不算可执行任务
如果任务只有方向，没有完成判定：
- 只能停留在分析阶段
- 不得声称“已完成”

### 4. 当前负责人不明确，不得继续推进
如果不知道当前是谁在负责：
- 不进入具体执行动作
- 先补 `current_owner` / `next_owner`

## 文档使用规则

### 1. 执行时的默认阅读顺序
1. `AGENTS.md`
2. `governance/task-schema.md`
3. `governance/decision-gates.md`
4. 对应 `workflows/playbooks/*.md`
5. 对应 `roles/playbooks/*.md`
6. `governance/handoff-templates.md`
7. 当前具体任务单 / 任务文档 / 样例文档

### 2. 文档冲突处理规则
- 项目级安全、审批、停止规则：以 `AGENTS.md` 为准
- 任务结构字段：以 `governance/task-schema.md` 为准
- 是否必须停机 / 等待拍板：以 `governance/decision-gates.md` 为准
- 当前任务怎么流转：以对应 workflow 为准
- 当前角色该做什么：以对应 role playbook 为准

### 3. 不得跳文档执行
禁止以下行为：
- 不看任务结构就直接开工
- 不看 workflow 就跳到局部执行
- 不看 decision gate 就越权推进
- 不写交接说明就切角色

## 项目级执行硬规则

### 1. 不得脑补缺失关键信息
如果文档不完整：
- 可以标记缺口
- 可以提出补充需求
- 不得把猜测写成确定结论

### 2. 每次推进必须有状态变化
每次有效推进至少要明确：
- 当前状态
- 当前负责人
- 当前产出
- 下一步接收方

没有状态变化，不算有效推进。

### 3. 每次交付必须附带说明
任何阶段产出都不能只写“已完成”。
至少要写：
- 做了什么
- 没做什么
- 风险点是什么
- 下一步建议是什么

### 4. 每次角色切换必须有交接说明
若发生角色切换，必须补：
- 已完成内容
- 未完成内容
- 已知风险
- 下一角色先看什么 / 先做什么

没有交接说明，不算完成流转。

### 5. 返工必须具体
返工意见不得模糊。
必须具体到：
- 哪个点不达标
- 对应哪条验收标准
- 应改成什么
- 下次重点复查什么

### 6. 不得越权把高风险动作当常规动作推进
涉及生产、外部影响、删除、付款、对外承诺等行为时，不得自行越权推进。

---

## 必须停止并进入 `Waiting Decision` 的情况
出现以下任一情况，必须停下：
- 目标不明确
- 需求互相冲突
- 缺少关键上下文
- 多个方案并存且缺少决策标准
- 涉及外部影响
- 涉及高风险操作
- 审核未通过但仍想强推
- hotfix 涉及回滚、降级、生产数据修复、正式对外说明

进入 `Waiting Decision` 时，必须同时输出：
- 当前结论
- 为什么不能继续自动推进
- 可选方案
- 风险差异
- 推荐选项
- 等待谁决策

## 文档编写硬要求

### 1. 所有执行型文档都应尽量包含
- 目标
- 前置条件
- 输入
- 步骤
- 输出
- 验收标准
- 异常处理
- 交接对象

### 2. 避免空话
少写：
- 合理推进
- 做好质量把控
- 视情况处理

多写：
- 检查哪些字段
- 执行哪些动作
- 什么条件才算完成
- 不满足时退回哪里

### 3. 优先结构化表达
- 能用字段就不用散文
- 能用清单就不用长段描述
- 能用模板就不要每次重写

### 4. 一个文档只解决一类问题
避免一份文档同时承担：
- 项目介绍
- 角色说明
- workflow 细节
- 执行细则

如果职责混乱，代理就更容易读歪。

---

## 当前阶段的成功标准
当前阶段不是追求“最强代理”，而是追求：
- 能起任务
- 能选 workflow
- 能按角色推进
- 能在关键节点停下
- 能完成交接
- 能完成收口
- 能把同一方法复制到后续场景

---

## 一句话判断标准
如果当前文档和任务状态，不能让 Codex 明确知道：
- 现在该做什么
- 做完产出什么
- 交给谁
- 什么情况下必须停

那就说明还不具备可执行性，不得假装进入正式执行。
