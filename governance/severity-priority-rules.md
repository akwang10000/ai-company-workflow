# governance/severity-priority-rules.md

## 目标
定义研发团队版在 **bug / incident / hotfix** 场景下统一使用的严重度（Severity）与优先级（Priority）判断口径，避免：
- 同一个问题被不同角色用不同标准描述
- 问题很严重，但任务单里只写了普通 bugfix
- 实际只是小缺陷，却被误判成 hotfix
- PM、Tech Lead、QA、Ops 对“先修哪个”没有统一语言

这份文档解决两个问题：
1. **这个问题有多严重**（Severity）
2. **这个问题要多快处理**（Priority）

---

## 先讲结论
- **Severity**：看影响有多大、后果有多重
- **Priority**：看现在多急、是否要立刻插队处理
- 严重度高，不一定优先级永远最高
- 优先级高，也不一定说明问题本身最严重

例如：
- 一个线上支付全挂的问题：Severity 高，Priority 也高
- 一个只影响极少数内部账号的权限显示问题：Severity 可能中，但如果 CEO 正在演示，Priority 也可能临时拉高

---

## 核心原则

### 1. 先判断影响，再判断紧急度
不要一上来就说“高优先级”。
先看：
- 影响谁
- 影响多少人
- 是否影响核心链路
- 是否造成数据错误 / 数据丢失 / 对外事故

再看：
- 是否正在发生
- 是否有临时绕过方案
- 是否卡住上线 / 交付 / 客户承诺
- 是否必须马上抢修

### 2. 严重度与优先级必须分开写
任务单里：
- `severity` 负责表达问题严重程度
- `priority` 负责表达处理顺序

不能只写一个“high”就糊过去。

### 3. 先保守，不要滥用最高级别
当前项目还在第一阶段，默认宁可多触发人工判断，也不要把所有问题都标成最高级别。

### 4. 进入 hotfix 的门槛要明确
不是所有线上 bug 都是 hotfix。
满足事故级口径，或明确需要打断当前计划抢修，才进入 hotfix 流程。

---

## Severity 定义（问题严重度）

建议统一使用四级：
- `S0`：事故级 / 灾难级
- `S1`：严重
- `S2`：中等
- `S3`：轻微

---

### `S0`：事故级 / 灾难级
#### 定义
问题已经造成或正在造成重大业务中断、重大数据风险、重大外部影响。

#### 常见判定条件（命中任一即可优先按 S0 处理）
- 核心主链路不可用
- 大面积用户无法使用关键功能
- 发生数据丢失、数据破坏、严重数据错乱
- 涉及资金、订单、权限、安全等重大风险
- 正在造成对外事故，且无法通过简单绕过止血
- 需要紧急回滚、降级、封禁、停服、生产修复

#### 例子
- 登录服务整体不可用
- 支付成功但订单未入账
- 用户看到他人隐私数据
- 线上错误脚本正在持续覆盖生产数据

#### 默认动作
- 优先进入 `hotfix`
- 立即触发 `Waiting Decision` 或事故流程中的人工审批门
- 当前负责人需立刻升级到 Tech Lead / Ops / Approver 联动

---

### `S1`：严重
#### 定义
问题未达到灾难级，但已明显影响核心功能、关键用户、关键交付，且不能长期放置。

#### 常见判定条件
- 核心功能部分不可用
- 影响较大范围用户，但未全量中断
- 存在明显错误结果，短期内会造成业务损失或高投诉
- 没有安全事故级风险，但继续放任会快速扩大影响
- 关键版本发布被阻塞

#### 例子
- 核心查询接口在某主流程中 50% 失败
- 管理后台无法执行关键运营动作
- 某主版本上线前发现会导致主要业务流程失败

#### 默认动作
- 通常进入 `bugfix`，必要时升级为 `hotfix`
- 当日优先处理
- 需要明确 owner、影响范围、临时缓解方案

---

### `S2`：中等
#### 定义
问题真实存在，也影响质量或效率，但不构成事故级风险，通常存在替代方案或绕过路径。

#### 常见判定条件
- 非核心功能异常
- 只影响特定场景 / 特定角色 / 小范围用户
- 有可接受的人工绕过方式
- 不会立刻造成更大事故

#### 例子
- 某筛选条件无效，但用户还能换条件完成查询
- 某后台页面布局错乱，但数据仍可操作
- 导出功能缺一个次要字段

#### 默认动作
- 一般走 `bugfix`
- 按版本节奏排期
- 不默认打断当前高优先级工作

---

### `S3`：轻微
#### 定义
问题存在，但主要影响体验、文案、边角逻辑或低频场景，不影响主链路可用性。

#### 常见判定条件
- 文案错误
- 样式错位
- 低频场景表现不一致
- 对主要结果无实质影响

#### 例子
- 按钮文案错误
- tooltip 描述过期
- 空状态图不正确

#### 默认动作
- 纳入普通排期
- 不进入 hotfix
- 可与同类小问题合并修复

---

## Priority 定义（处理优先级）

建议统一使用四级：
- `P0`：立即处理
- `P1`：优先处理
- `P2`：正常排期
- `P3`：可延后

---

### `P0`：立即处理
#### 定义
必须立刻插队处理，暂停或压缩其他工作。

#### 典型触发条件
- 线上事故正在发生
- 不立即处理会持续扩大损失
- 已阻断当天关键交付 / 上线 / 客户承诺
- 没有可接受的绕过方案

#### 默认要求
- 立即指定 owner
- 明确临时止血方案
- 明确下一次状态更新时间
- 必要时走 hotfix

---

### `P1`：优先处理
#### 定义
不一定要分钟级抢修，但应尽快处理，通常应放在当前迭代的前列。

#### 典型触发条件
- 严重 bug，但已有临时绕过方案
- 阻塞重要版本，但还未形成事故
- 影响关键客户 / 核心业务角色

---

### `P2`：正常排期
#### 定义
需要处理，但按正常节奏安排即可。

#### 典型触发条件
- 中等影响问题
- 有替代方案
- 不阻塞关键里程碑

---

### `P3`：可延后
#### 定义
短期不处理也可接受，适合纳入后续优化或合并修复。

#### 典型触发条件
- 体验类小问题
- 低频问题
- 不影响交付与主流程

---

## Severity 与 Priority 的推荐映射

| Severity | 默认 Priority | 默认 workflow | 说明 |
|---|---|---|---|
| S0 | P0 | hotfix | 事故级，默认抢修 |
| S1 | P1 / P0 | bugfix 或 hotfix | 看是否正在扩大、是否有绕过 |
| S2 | P2 | bugfix | 正常缺陷处理 |
| S3 | P3 / P2 | bugfix | 体验或低频问题 |

### 注意
以下情况允许“提优先级，不提严重度”：
- 正在演示 / 对外汇报
- 卡住今天必须交付的版本
- 关键客户正在使用
- 高层明确要求优先处理

但提优先级时，必须在任务单里写明原因，不能偷改 severity。

---

## 何时从 bugfix 升级为 hotfix
命中以下任一情况，可优先考虑升级为 `hotfix`：
- Severity = `S0`
- Priority = `P0`
- 问题正在生产环境持续发生
- 需要立即回滚 / 降级 / 热修
- 不修会持续扩大损失
- 已出现明确外部影响，不能等正常排期

如果只是：
- 线上有 bug，但影响很小
- 有明确绕过方案
- 不会持续扩大
- 不影响关键承诺

则仍优先按 `bugfix` 处理，不要滥开 hotfix。

## 研发团队版判断流程（推荐）

### 第一步：判断 Severity
按以下顺序问：
1. 是否影响核心主链路？
2. 是否影响大量用户或关键客户？
3. 是否涉及数据、资金、权限、安全？
4. 是否已经造成外部事故？
5. 是否只是体验 / 局部问题？

### 第二步：判断 Priority
继续问：
1. 是否正在发生、必须立刻止血？
2. 是否有临时绕过方案？
3. 是否卡住上线 / 交付 / 演示 / 承诺？
4. 是否必须打断当前任务？

### 第三步：判断 workflow
- `S0` 或 `P0`：优先检查是否进入 `hotfix`
- 其他：默认进入 `bugfix`

### 第四步：判断是否触发 `Waiting Decision`
若涉及：
- 回滚
- 生产数据修复
- 外部公告
- 高风险配置修改
- 多种止血方案需取舍

则必须进入 `Waiting Decision`。

## 任务单字段建议补充
当前 `task-schema.md` 已有：
- `priority`
- `risk_level`

建议在 bug / incident / hotfix 场景下补充以下字段：
- `severity`
- `impact_scope`
- `user_impact`
- `workaround`
- `incident_started_at`
- `target_resolution_time`

### 建议写法
```yaml
severity: S1
priority: P0
impact_scope: 核心后台查询链路，影响全部运营账号
user_impact: 运营无法自助查询封禁记录，当前需研发人工查库
workaround: 暂时由研发手工查库支持，但效率低且不可持续
incident_started_at: 2026-03-29 08:40
target_resolution_time: 2026-03-29 11:00
```

## 分级示例

### 示例 1：支付回调失败导致订单状态不更新
- Severity：`S0`
- Priority：`P0`
- workflow：`hotfix`
- 原因：核心链路、业务直接受损、需要立即止血

### 示例 2：后台管理页主查询按钮失效
- Severity：`S1`
- Priority：`P1`
- workflow：`bugfix` 或 `hotfix`
- 原因：核心内部功能不可用；若今天正卡上线或大促，可升到 P0/hotfix

### 示例 3：导出报表缺少次要字段
- Severity：`S2`
- Priority：`P2`
- workflow：`bugfix`
- 原因：功能有缺陷，但通常可先用替代方案

### 示例 4：页面文案错字
- Severity：`S3`
- Priority：`P3`
- workflow：`bugfix`
- 原因：体验问题，不影响主流程

## 角色分工建议

### PM Agent
- 补齐影响描述
- 初判 priority
- 判断是否阻塞业务交付

### Tech Lead Agent
- 校验 severity 是否合理
- 判断是否需要 hotfix
- 给出技术风险与止血方案

### QA / Review Agent
- 复核影响范围是否被低估
- 检查回归范围与上线风险

### Ops / Release Agent
- 在 S0 / P0 场景下，确认发布窗口、回滚条件、值守要求

### CEO / 用户 / Approver
- 在多方案、对外影响、回滚取舍、生产数据修复等高风险决策上拍板

---

## 使用禁忌
- 不要把“老板催得急”直接写成高 Severity
- 不要把“看起来麻烦”直接升级成 hotfix
- 不要只写 Priority，不写 Severity
- 不要在没有影响证据时滥标 S0 / P0
- 不要因为是内部系统就低估生产影响

---

## 一句话判断标准
- **Severity 看后果有多重**
- **Priority 看现在多急**
- **hotfix 看是否必须打断正常节奏立刻止血**
