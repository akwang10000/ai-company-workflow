# governance/regression-checklist.md

## 目标
定义研发团队版在 **bugfix / hotfix / feature 交付前** 的最小回归检查口径，避免：
- 问题修了，但顺手把别的地方搞坏
- 只验证“我改的地方”，不验证受影响链路
- hotfix 抢修后没有留下最基本的回归记录
- QA、Dev、Tech Lead 对“回归做到什么程度才算够”没有统一标准

这份文档不是为了追求完美测试覆盖，
而是为了确保每次变更至少做到：
- 关键路径有检查
- 影响范围有边界
- 已验证与未验证说清楚

---

## 核心原则

### 1. 回归不是“再点一遍改过的按钮”
回归至少应覆盖：
- 修复点本身
- 修复点上下游主链路
- 最可能被这次改动波及的相邻功能

### 2. 回归范围要和风险匹配
- 小文案改动：不需要做整站回归
- 核心逻辑改动：不能只做局部冒烟
- hotfix：允许压缩，但不能没有回归

### 3. 已验证 / 未验证必须分开写
不能只写“已测试通过”。
必须说明：
- 验了什么
- 没验什么
- 为什么没验
- 剩余风险是什么

### 4. 有风险的变更，优先先做最小可证明回归
如果时间很紧：
- 先保住核心链路
- 再补边角场景
- 但不能用“来不及”替代记录

---

## 什么时候必须做回归清单
以下场景默认都要过这份清单：
- `bugfix`
- `hotfix`
- 影响已有功能的 `feature`
- 涉及权限、状态流转、数据写入、接口契约的改动

以下场景可适当简化：
- 纯文案修正
- 纯样式修正
- 明确不影响逻辑和数据的静态资源替换

即便简化，也应至少写明：
- 变更类型
- 为什么可简化
- 最小验证做了什么

## 最小回归清单（通用版）

### A. 变更理解
- [ ] 我已明确这次改动修的是什么问题 / 实现了什么目标
- [ ] 我已明确这次改动影响的是哪个模块 / 页面 / 接口 / 任务流转节点
- [ ] 我已明确这次改动是否涉及数据写入、权限判断、状态流转、外部依赖

### B. 修复点验证
- [ ] 原问题可以稳定复现，或已有足够证据说明原问题存在
- [ ] 修复后，原问题按预期消失
- [ ] 修复结果与任务单中的验收标准一致

### C. 相邻链路回归
- [ ] 上游输入路径仍可正常进入
- [ ] 下游结果 / 状态 / 数据输出仍正确
- [ ] 同模块下最接近的相邻功能未被误伤
- [ ] 关键异常路径（空值 / 无权限 / 非法输入 / 网络失败）至少抽查一项

### D. 数据与状态检查
- [ ] 如涉及写入，已确认不会产生重复写、错写、漏写
- [ ] 如涉及状态流转，已确认状态不会卡死、跳步、回退异常
- [ ] 如涉及查询展示，已确认展示字段与真实数据一致

### E. 权限与边界检查
- [ ] 有权限的人可以正常使用
- [ ] 无权限的人不会越权访问
- [ ] 边界条件下不会报错或 silent failure

### F. 发布前可交付性检查
- [ ] 已记录验证环境 / 数据条件 / 测试前提
- [ ] 已记录已验证范围
- [ ] 已记录未验证范围
- [ ] 已记录剩余风险
- [ ] 已给出是否可进入 `Ready for Delivery` 的明确结论

## 按任务类型补充的回归要求

### 一、bugfix 回归要求
除通用清单外，至少补查：
- [ ] 原 bug 场景已验证通过
- [ ] 同类入口至少抽查 1 个
- [ ] 最容易被连带影响的旧逻辑至少抽查 1 个
- [ ] 若修复涉及条件判断，已检查正向与反向条件

#### bugfix 输出建议
```md
## Regression Summary
- 已验证：
  - 原缺陷场景已修复
  - 相邻查询场景正常
  - 无权限访问被拒绝
- 未验证：
  - 大数据量压力场景未覆盖
  - 历史旧版本兼容性未覆盖
- 剩余风险：
  - 高并发下查询性能仍待线上观察
- 结论：
  - 可进入 Ready for Delivery
```

---

### 二、hotfix 回归要求
hotfix 允许压缩，但必须先保住止血有效和二次事故不扩散。

除通用清单外，至少补查：
- [ ] 止血动作本身已生效
- [ ] 事故主链路已恢复可用
- [ ] 回滚 / 降级 / 临时开关状态已确认
- [ ] 已定义上线后观察指标或观察窗口
- [ ] 已记录是否还需要 follow-up 修复任务

#### hotfix 额外要求
如果以下内容没法在抢修窗口完成，也必须明确写出：
- 哪些回归没有做
- 为什么这次先不上
- 上线后谁负责盯
- 何时补完整回归 / follow-up

#### hotfix 输出建议
```md
## Hotfix Regression Summary
- 已验证：
  - 核心故障链路恢复
  - 关键接口返回正常
  - 回滚开关状态确认无误
- 未验证：
  - 低频边缘入口未全量回归
  - 非核心后台页面未逐页检查
- 上线后观察：
  - 30 分钟重点观察错误率与核心接口耗时
- Follow-up：
  - 需要补一条完整 bugfix 任务清理临时方案
- 结论：
  - 可发布，但需值守观察
```

---

### 三、feature 回归要求
如果 feature 改动会碰到已有逻辑，除了验证新功能本身，还要确认旧功能没被带坏。

除通用清单外，至少补查：
- [ ] 新功能主路径可走通
- [ ] 老功能原有主路径仍可走通
- [ ] 若涉及表单 / 查询 / 写入，已抽查正常输入与边界输入
- [ ] 若涉及权限，已抽查至少一组有权 / 无权账号

## 回归范围分层（推荐）

### Level 1：局部验证
适用于：
- 文案 / 样式 / 非核心展示调整

至少应覆盖：
- 修复点本身
- 最近相邻页面 / 组件 1 处

### Level 2：模块回归
适用于：
- 一般 bugfix
- 非事故级功能改动

至少应覆盖：
- 修复点
- 上下游主路径
- 1~2 个高概率受影响场景

### Level 3：核心链路回归
适用于：
- 核心业务逻辑改动
- 权限、状态流转、数据处理改动
- 高风险版本发布前

至少应覆盖：
- 修复点
- 主链路全路径
- 关键异常路径
- 关键权限边界

### Level 4：事故后重点回归
适用于：
- S0 / S1 的 hotfix 或重大事故修复

至少应覆盖：
- 事故主链路恢复
- 邻近高风险链路
- 回滚条件 / 开关状态
- 发布后观察计划

---

## 谁负责什么

### Dev Agent
- 先做自测回归
- 不得只说“本地 OK”
- 必须明确已验证范围与未验证范围

### QA / Review Agent
- 复核回归范围是否过窄
- 抽查高风险路径是否遗漏
- 给出是否进入 `Ready for Delivery` 的判断

### Tech Lead Agent
- 对高风险改动，确认回归层级是否足够
- 判断是否需要扩大验证范围或延后交付

### Ops / Release Agent
- 对 hotfix / 高风险发布，确认上线观察项、回滚条件和值守安排

---

## 常见错误
- 只测 happy path，不测异常路径
- 只测自己改的函数，不测业务流
- 只说“测试通过”，不说测了哪些
- 抢修后直接上线，不写观察计划
- 明明没测权限，却默认权限没问题
- 明明没测数据写入，却说“应该没影响”

## 最小回归记录模板

```md
# Regression Checklist

## 任务
- task_id:
- title:
- task_type:
- owner:

## 本次变更
- 修复 / 新增内容：
- 涉及模块：
- 风险点：

## 已验证
- [ ] 修复点验证通过
- [ ] 上游输入路径验证通过
- [ ] 下游结果验证通过
- [ ] 相邻功能抽查通过
- [ ] 权限 / 边界抽查通过

## 未验证
- 

## 剩余风险
- 

## 结论
- [ ] 可进入 Ready for Delivery
- [ ] 需继续补回归
- [ ] 需进入 Waiting Decision
```

---

## 一句话判断标准
如果当前产出不能清楚回答：
- 修复点验没验
- 相邻链路验没验
- 哪些没验
- 风险还剩什么

那就说明这次回归还不算完整。
