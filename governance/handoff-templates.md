# governance/handoff-templates.md

## 目标
定义角色之间的标准交接模板，避免任务在多智能体流转时出现：
- 上下文丢失
- 责任不清
- 返工理由模糊
- 下一步接手方不知道先看什么

这份文档的核心作用是：
> 让 Codex / 智能体在每次交接时，都用统一格式表达“我做到哪一步了，接下来该你做什么”。

---

## 全局交接原则

### 1. 没有交接说明，不算完成流转
任务状态变化时，必须附带 `handoff_note` 或等价交接内容。

### 2. 交接不是复述全部上下文，而是提炼执行关键信息
交接重点不是“把所有东西再说一遍”，而是告诉下一角色：
- 当前到底完成了什么
- 哪些东西最关键
- 哪些坑已经发现
- 你下一步先干什么

### 3. 交接必须服务下一步执行
如果一份交接说明不能让接手方马上开工，那这份交接说明就不合格。

### 4. 风险和未完成项必须显式写出
不允许把风险藏在长文里，也不允许假装任务已经没有问题。

---

## 通用交接模板

### 标准结构
- `from_role`：交出方
- `to_role`：接收方
- `task_id`：任务 ID
- `task_title`：任务名称
- `current_status`：当前状态
- `completed_work`：已完成内容
- `key_context`：关键信息
- `artifacts`：相关产物
- `known_risks`：已知风险
- `open_questions`：待解问题
- `next_action`：建议下一步动作
- `handoff_ready_check`：是否满足交接条件

---

## 通用交接模板示例

```yaml
from_role: PM Agent
to_role: Tech Lead Agent
task_id: ENG-001
task_title: 新增用户封禁记录查询页
current_status: In Analysis
completed_work:
  - 已明确业务目标
  - 已确认查询字段范围
  - 已定义基础验收标准
key_context:
  - 页面只允许查询，不允许修改
  - 仅运营主管与客服主管可访问
  - 目标是降低研发人工查库支持成本
artifacts:
  - docs/tasks/ENG-001-requirement.md
known_risks:
  - 历史数据字段完整性未知
open_questions:
  - 是否需要分页
  - 是否需要导出功能
next_action:
  - 请完成技术方案评估，并标注接口、权限、性能风险
handoff_ready_check: true
```

## 研发团队版常用交接模板

## 1. CEO / 用户 -> PM Agent
### 适用场景
业务目标刚提出，需要转为可执行需求。

### 模板
- 任务目标：
- 为什么要做：
- 当前已知背景：
- 明确不做什么：
- 截止时间 / 优先级：
- 需要 PM 先澄清的问题：

### 完成交接标准
- PM 能根据输入继续写需求摘要
- 目标、背景、边界至少有初始描述

---

## 2. PM Agent -> Tech Lead Agent
### 适用场景
需求已初步明确，进入技术评估。

### 模板
- 已确认需求：
- 验收标准：
- 业务边界：
- 当前疑点：
- 需要重点评估的技术问题：
- 相关文档 / 路径：

### 完成交接标准
- Tech Lead 能独立开始方案评估
- 不需要回头重新问“到底要做什么”

---

## 3. Tech Lead Agent -> Dev Agent
### 适用场景
技术方案已定，进入开发执行。

### 模板
- 方案摘要：
- 建议实现路径：
- 模块拆分：
- 风险点：
- 不允许偏离的约束：
- 需要保留的日志 / 注释 / 说明：
- Dev 开工前必须确认事项：

### 完成交接标准
- Dev 能明确知道改哪里、为什么改、不能碰什么

---

## 4. Dev Agent -> QA / Review Agent
### 适用场景
开发完成，进入审核。

### 模板
- 已完成实现：
- 修改点列表：
- 自测范围：
- 自测结果：
- 未覆盖项：
- 已知风险：
- 请 Review 重点检查：

### 完成交接标准
- QA / Review 不需要重新猜这次改动到底改了什么
- 自测与未测边界清楚

---

## 5. QA / Review Agent -> Dev Agent（返工）
### 适用场景
审核未通过，退回返工。

### 模板
- 驳回原因：
- 不符合项清单：
- 每项对应修改要求：
- 建议优先级：
- 返工后重点复查项：

### 完成交接标准
- Dev 能直接按问题列表修，而不是自己猜

---

## 6. QA / Review Agent -> CEO / 用户
### 适用场景
审核通过或存在需拍板风险，等待最终决定。

### 模板
- 当前结论：通过 / 有条件通过 / 不建议上线
- 通过了什么：
- 剩余风险：
- 建议动作：
- 是否建议立即交付：

### 完成交接标准
- CEO / 用户能快速判断要不要继续推进

---

## 7. CEO / 用户 -> Ops / Release Agent
### 适用场景
任务已获批准，准备收口或发布说明。

### 模板
- 最终决策：
- 允许推进的范围：
- 当前限制：
- 需要记录的重点：
- 是否归档 / 发布 / 跟踪：

### 完成交接标准
- Ops 能直接产出交付摘要或发布说明

---

## 返工交接特殊规则
返工时必须额外写清：
- 本次返工属于哪一轮
- 上一轮结论是什么
- 返工目标是什么
- 若再失败，是否升级人工判断

不允许使用这些废话式交接：
- 再优化一下
- 这里有点问题
- 看着改改
- 体验不太对

必须改成：
- 字段 `ban_reason` 未展示，不满足验收标准第 3 条
- 无权限账号访问接口未被拦截，不满足安全要求

---

## Codex 执行要求
当 Codex 负责某一角色时，交接前必须自检：
- 我是否说明了已完成内容？
- 我是否说明了未完成内容？
- 我是否标记了风险？
- 我是否指出了下一角色先做什么？
- 我是否提供了必要产物路径或引用？
- 如果任务还不能交接，我是否诚实说明了原因？
