# governance/ready-for-delivery-checklist.md

## 目标
定义任务在进入 `Ready for Delivery` 前必须完成的最小收口检查，避免：
- 代码/文档刚改完就自称可交付
- 角色之间以为“差不多了”，但没人真正收口
- QA 觉得还没稳，Dev 已经往外推
- 交付时缺少风险说明、已知限制、产物索引

这份文档解决的问题是：
> **什么情况下，一个任务才真的具备“可以交付给用户 / CEO / Approver 做最终确认”的条件。**

---

## 核心原则

### 1. `Ready for Delivery` 不是 `Done`
`Ready for Delivery` 的含义是：
- 主要工作已完成
- 产物已具备可审阅 / 可验收条件
- 风险和缺口已被说清楚
- 现在可以交给最终确认人拍板

它不等于：
- 已正式上线
- 已被用户确认通过
- 已完成归档

### 2. 没有收口说明，不算 Ready
不能只改完东西就把状态切到 `Ready for Delivery`。
至少要能回答：
- 做了什么
- 怎么验证的
- 还有什么没做
- 有什么风险
- 交给谁确认

### 3. 风险可带着走，但不能藏着走
允许带着已知风险进入 `Ready for Delivery`，
但必须满足：
- 风险已被记录
- 风险未超出允许范围
- 最终确认人知道并接受

### 4. 缺少关键验证时，不能硬进 Ready
如果缺少：
- 基本验收结果
- 基本回归记录
- 当前版本产物说明
- 明确的交付对象

则不得进入 `Ready for Delivery`。

## 进入 `Ready for Delivery` 的最小检查清单

### A. 任务定义完整性
- [ ] 任务单存在且可读
- [ ] `title / goal / status / current_owner / acceptance_criteria` 已补齐
- [ ] 当前任务范围没有明显漂移或未决冲突
- [ ] 当前任务不处于 `Waiting Decision`

### B. 产出完整性
- [ ] 本次承诺交付的主要产物已经产出
- [ ] 产物位置清楚可查（代码、文档、报告、截图、链接等）
- [ ] 没有把“未来再补”的关键项伪装成已完成

### C. 验收与验证
- [ ] 至少已对照 `acceptance_criteria` 做过检查
- [ ] 已有可供审阅的验证结果
- [ ] 若属于 bugfix / hotfix / 高风险 feature，已附回归结论
- [ ] 已明确哪些范围已验证，哪些尚未验证

### D. 风险与限制
- [ ] 已记录已知风险
- [ ] 已记录已知限制 / 未覆盖项
- [ ] 若存在临时方案，已明确是否需要 follow-up
- [ ] 若风险超出自动放行范围，已转入 `Waiting Decision`

### E. 交付说明
- [ ] 已明确交付对象（CEO / 用户 / Approver / Ops）
- [ ] 已写清“做了什么、没做什么、建议怎么验收”
- [ ] 已说明下一步由谁接收
- [ ] 若需要上线 / 发布，已说明是否还需发布动作

## 标准交付摘要模板
进入 `Ready for Delivery` 时，建议至少附以下摘要：

```md
## Delivery Summary
- 任务：
- 当前状态：Ready for Delivery
- 当前负责人：
- 已完成：
- 未完成：
- 已验证：
- 未验证：
- 已知风险：
- 建议验收方式：
- 下一接收方：
```

## 按任务类型补充检查

### 一、feature
除通用清单外，还应确认：
- [ ] 新功能主路径可演示 / 可验证
- [ ] 关键交互说明已补齐
- [ ] 若涉及权限 / 配置 / 开关，已写清使用前提
- [ ] 若只完成第一版，已明确非目标范围

### 二、bugfix
除通用清单外，还应确认：
- [ ] 原问题已修复
- [ ] 回归范围已写明
- [ ] 若仍有剩余风险，已说明可接受边界

### 三、hotfix
除通用清单外，还应确认：
- [ ] 止血目标已达成
- [ ] 是否还需要 follow-up 任务已明确
- [ ] 是否需要上线后观察已明确
- [ ] 是否需要 postmortem 已明确

### 四、research / proposal
除通用清单外，还应确认：
- [ ] 结论、假设、建议分开写清楚
- [ ] 未验证项没有伪装成结论
- [ ] 已明确推荐路线或下一步动作

---

## 不允许进入 `Ready for Delivery` 的典型情况
出现以下任一情况，不得把状态切到 `Ready for Delivery`：
- 验收标准还没对齐
- 当前还有关键阻塞未解
- 风险还没说清楚
- 修复点还没验证
- 回归完全没做
- 当前负责人和下一接收方都不明确
- 还在等拍板，却假装已可交付

---

## 谁可以把状态切到 `Ready for Delivery`
建议默认由以下角色切换：
- `QA / Review Agent`
- `Tech Lead Agent`
- 在小任务中，也可由明确授权的 `Dev Agent` 切换

但无论谁切，都必须附带：
- 交付摘要
- 验证结论
- 风险说明

不能裸切状态。

---

## 与 `Done` / `Archived` 的区别

### `Ready for Delivery`
表示：
- 具备交付条件
- 等待最终确认 / 上线 / 接收

### `Done`
表示：
- 已完成最终确认
- 已满足 `definition_of_done`

### `Archived`
表示：
- 已完成归档
- 结果、产物、复盘信息可回查

## 一句话判断标准
如果当前任务还不能让接收方清楚知道：
- 现在拿到的是什么
- 它已经被验证到什么程度
- 还剩什么风险
- 我下一步该怎么确认

那它就还不算 `Ready for Delivery`。
