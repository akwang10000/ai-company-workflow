# governance/decision-gates.md

## 目标
定义项目中的人工决策门（Decision Gates），避免 Codex / 智能体在关键节点越权推进。

---

## 核心原则
- 低风险动作可按文档自动推进
- 高风险动作必须停下
- 停下时不能只说“需要你确认”
- 必须同时提供：
  - 当前结论
  - 可选方案
  - 风险差异
  - 推荐选项
  - 等待谁决策

---

## 必须进入 `Waiting Decision` 的情况

### 1. 目标不明确
例如：
- 需求目标互相冲突
- 用户只给方向，没有完成标准
- 任务范围持续漂移

### 2. 多方案并存且缺少决策标准
例如：
- 技术实现有两条路线
- 短期快做与长期可维护性冲突
- 方案成本差距较大

### 3. 涉及外部影响
包括但不限于：
- 对外发布
- 对外承诺
- 联系客户 / 用户 / 合作方
- 正式发文 / 发帖 / 上线

### 4. 涉及高风险操作
包括但不限于：
- 删除数据
- 删除文件
- 生产环境变更
- 付款
- 权限批量修改
- 高风险配置变更

### 5. 风险超过文档允许范围
例如：
- 缺少关键上下文仍想推进
- 变更影响范围不清楚
- 审核未通过但希望强推交付

## 决策请求模板
当任务进入 `Waiting Decision` 时，必须按以下结构输出：

### 1. 当前任务
- 任务名：
- 当前状态：Waiting Decision
- 当前负责人：

### 2. 为什么停下
- 决策原因：
- 不继续自动推进的原因：

### 3. 可选方案
- 方案 A：
  - 优点：
  - 风险：
- 方案 B：
  - 优点：
  - 风险：

### 4. 推荐意见
- 推荐方案：
- 推荐原因：

### 5. 等待决策人
- Approver：CEO / 用户 / 指定负责人

## 研发团队版常见决策门

### Gate 1：需求范围确认
触发条件：
- PM 无法从当前输入中确认边界

### Gate 2：技术方案选型
触发条件：
- Tech Lead 给出多个方案且差异明显

### Gate 3：高风险实现
触发条件：
- 影响生产数据
- 涉及重大兼容性风险

### Gate 4：发布前确认
触发条件：
- 功能已完成，但是否上线需要业务确认

### Gate 5：审核未通过但业务想强推
触发条件：
- QA / Review 标注存在明确风险
- 业务仍要求推进

---

## 默认推荐
当前项目阶段建议：
- 审批多一点，先保守
- 等流程跑顺后，再逐步放开白名单自动执行项

先别追求“全自动”，先追求“可控自动化”。
