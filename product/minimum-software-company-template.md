# product/minimum-software-company-template.md

## 目标
定义第一阶段“第一个软件公司模板”的最小可运行边界，确保产品、前后端、模板初始化流程、以及 Codex 的实施顺序都围绕同一套默认骨架推进。

这份文档主要回答：
- 第一个软件公司模板到底包含什么
- 用户从模板启动后会得到什么
- 哪些内容是默认预置，哪些留给用户后调
- 第一阶段为什么先做这个最小模板，而不是一上来做通用模板市场

---

## 模板定位
“第一个软件公司模板”不是最终行业模板全集，而是 AI Company Workflow 第一阶段的验证样板。

它的定位是：
- 先验证研发团队场景能不能跑通
- 先验证任务流转、角色协作、交接、审批、收口是否闭环
- 先让用户不需要从空白开始搭系统
- 先为后续多行业模板复用提供母版结构

一句话说：
它不是“最全模板”，而是**第一个能启动、能跑任务、能交付、能复用的模板母版**。

---

## 设计原则

### 1. 模板优先，不从空白起步
第一阶段默认推荐用户从模板启动，而不是先创建空白 workspace 再手工配置一切。

### 2. 先可运行，再可定制
用户启动后应先拿到一套能工作的系统骨架，再根据需要微调角色、技能、说明、流程节点。

### 3. 先覆盖主链路，不追求全变体
第一阶段优先覆盖研发团队的 feature / bugfix / hotfix 主链路，不急着穷举边角场景。

### 4. 先保证交接和停机，再追求高度自治
模板重点是让任务能被接住、交出去、暂停决策、收口交付；不是先追求无限自动化。

---

## 默认包含的角色
模板默认至少包含以下角色模板：

### 1. PM Agent
负责：
- 需求澄清
- 验收标准定义
- 任务拆解
- 与用户 / 创业者对齐目标

### 2. Tech Lead Agent
负责：
- 技术方案判断
- 风险识别
- 模块拆分
- 实施边界确认

### 3. Dev Agent
负责：
- 功能实现
- 缺陷修复
- 自测与变更说明输出

### 4. QA / Review Agent
负责：
- 审核实现结果
- 回归检查
- 返工意见输出

### 5. Ops / Release Agent
负责：
- 交付摘要整理
- 发布说明输出
- hotfix / 事故后收口辅助

---

## 默认包含的 workflow
模板默认至少包含三套 workflow 模板：

### 1. feature workflow
用于常规功能需求。

主路径：
PM -> Tech Lead -> Dev -> QA / Review -> Ready for Delivery -> Done

### 2. bugfix workflow
用于常规缺陷修复。

主路径：
PM 或 Tech Lead 录入 -> Tech Lead 判断 -> Dev 修复 -> QA 回归 -> Ready for Delivery -> Done

### 3. hotfix workflow
用于线上高优先级问题止血。

主路径：
快速确认 -> Tech Lead 判断 -> Dev 热修 -> QA 关键路径验证 -> Ops 收口 -> Done / Postmortem

说明：
- 三套 workflow 都要支持返工回流
- 都要支持进入 `Waiting Decision`
- 都要支持必要的 handoff / review / decision 留痕

---

## 默认包含的任务样例
模板默认提供至少三类任务样例：
- `feature-example`
- `bugfix-example`
- `hotfix-example`

样例的作用不是替代真实任务，而是：
- 帮助用户理解任务单怎么写
- 帮助系统提供新建任务时的预填结构
- 帮助 Codex / 智能体明确最低输入要求

第一阶段建议任务样例至少覆盖：
- 标题
- 背景
- 目标
- 范围
- 验收标准
- 风险说明
- 当前状态
- 当前负责人
- 交付物要求

---

## 默认技能包
模板默认随角色挂载一组基础技能包，具体映射以 `product/role-skill-mapping.md` 为准。

第一阶段要求：
- 每个默认角色至少有一组基础技能包
- workflow 节点详情页能看到当前阶段相关技能
- 技能默认启用，但允许后续微调
- 技能系统先做到“可见、可挂载、可理解”，不急着做复杂编排

模板层面只需要保证：
- 角色和技能的默认关系明确
- 实例化后技能能跟随角色一起落地
- 节点可以根据 workflow 类型展示相关技能说明

---

## 默认首页 / 仪表盘
模板实例化后，workspace 首页至少应包含：
- 当前任务数
- 按状态统计的任务概览
- 待决策数量
- Ready for Delivery 数量
- 最近活动 / 最近交接
- 推荐第一步操作入口

推荐快捷入口：
- 新建任务
- 查看 workflow 画布
- 查看角色与技能
- 查看模板说明

首页目标不是做 BI，而是让用户一进来就知道：
- 现在系统有没有在跑
- 卡在哪
- 下一步干什么

---

## 模板实例化后的默认结果
用户点击“从模板启动”后，系统默认至少生成：
- 1 个 `Workspace`
- 1 组默认 `RoleAssignment`
- 3 套可运行 workflow 骨架
- 3 类任务样例入口
- 1 套基础 dashboard 数据结构
- 1 组默认技能挂载结果

如果第一阶段实现完整，用户启动后的体验应是：
- 不需要自己先定义角色
- 不需要自己先手工画流程
- 不需要自己先发明任务字段
- 可以直接创建第一条 feature / bugfix / hotfix 任务

---

## 用户可改项 / 系统保留项

### 用户可改项
启动后允许用户逐步修改：
- workspace 名称
- 模板实例显示名称
- 角色显示名称
- 角色启用 / 禁用状态
- 默认技能包启停
- workflow 描述与非关键说明文本
- 自定义补充任务模板

### 系统保留项
第一阶段建议作为系统母版能力保留：
- 核心状态集合
- 主 workflow 骨架结构
- 默认 decision / handoff / review 记录机制
- Ready for Delivery / Done 的基础门槛
- 系统关键字段命名与 API 主结构

理由：
如果这些核心骨架也在第一阶段放任自由编辑，系统会很快失去模板复用意义，Codex 也难以稳定施工。

---

## 第一阶段不做什么
为了避免第一版失控，当前模板不优先做：
- 多行业通用模板市场
- 高复杂度角色权限系统
- 节点级超细粒度动态技能编排
- 深度 BI / 分析大屏
- 大规模自动化调度与多工作区治理体系
- 用户从零自由拼装全部流程节点的高级编辑器

第一阶段要先把“默认模板启动后能顺畅跑起来”做好，再考虑高级玩法。

## 对领域模型的要求
为了支撑这个最小模板，领域模型至少要能表达：
- `CompanyTemplate`
- `RoleTemplate`
- `WorkflowTemplate`
- `TaskTemplate`
- `SkillPackageTemplate`
- `Workspace`
- `RoleAssignment`
- `TaskInstance`
- `WorkflowInstance`（或等价运行态结构）

其中关键关系是：
- `CompanyTemplate` 包含多个角色模板、workflow 模板、任务模板
- 实例化后生成 workspace 内的角色分配与 workflow 骨架
- 任务再基于 workflow 骨架进入运行态

---

## Codex 实施建议
如果让 Codex 基于本文档推进实施，建议顺序：

1. 先固化模板种子数据
   - 先把默认角色、workflow、任务样例、技能映射固化成种子数据或配置

2. 再打通模板实例化主链路
   - 支持从 `CompanyTemplate` 创建 `Workspace`
   - 落默认角色和 workflow 骨架

3. 再补首页与任务入口
   - 让用户能看到仪表盘、创建第一条任务、进入任务详情

4. 再补模板微调能力
   - 支持角色显示名修改、技能启停、模板说明修改

5. 最后再做更灵活的模板扩展
   - 比如多行业复制、模板变体、LLM 辅助生成模板草稿

---

## 验收标准
当第一阶段“第一个软件公司模板”达到可用状态时，至少应满足：
- 用户可以从模板一键启动 workspace
- 系统自动生成默认角色与 workflow 骨架
- 用户可以直接创建 feature / bugfix / hotfix 任务
- 首页能看到任务概览、待决策、待交付信息
- 节点详情能看到相关角色与技能
- 任务流转受到统一状态守卫约束
- Codex 能按阶段继续实现前后端功能，而不是每轮重新猜产品结构

---

## 一句话总结
“第一个软件公司模板”的任务，不是证明系统有多万能，而是先把**一个能启动、能分工、能流转、能停机、能交付的软件公司骨架**做扎实。
