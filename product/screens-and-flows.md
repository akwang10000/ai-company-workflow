# product/screens-and-flows.md

## 目标

定义 AI Company Workflow 第一阶段的主要页面和页面流转，确保前端实现时：

- 知道有哪些页面
- 知道每个页面解决什么问题
- 知道页面之间怎么跳转
- 知道 PC 和手机的职责分工
- 知道如何把“系统内部复杂、用户使用简单”落实到页面结构

---

这份文档负责：

- 页面清单

- 页面间跳转关系
- 页面层的体验约束

## 页面体验总原则

第一阶段的页面设计，要优先服务这几件事：

- 用户不用先理解内部对象模型，就能开始操作
- 用户一进入页面就能知道当前任务 / 当前卡点 / 下一步动作
- 默认页面只显示完成当前任务所需的信息
- 高复杂度信息进入二级页、详情区、抽屉或高级层

### 用户可见语言原则

页面默认应优先使用用户动作语言，例如：

- 从模板启动
- 标记为可开始
- 开始执行
- 提交交接
- 接收交接
- 请求拍板
- 驳回返工
- 标记可交付
- 完成交付

不应默认把这些内部术语直接作为主文案：

- transition
- payload
- guard
- workflow instance
- role assignment

### 信息暴露原则

默认页面优先展示：

- 现在是什么
- 卡在哪里
- 谁在负责
- 下一步做什么

默认不优先展示：

- 全量内部字段
- 全量记录原文
- 全量技能参数
- 复杂配置项

## UI 文案映射（Phase 1 默认口径）

### 状态文案映射

| 系统状态 | 默认用户可见文案 |
|---|---|
| `Draft` | 草稿 |
| `Ready` | 可开始 |
| `In Progress` | 进行中 |
| `Waiting Handoff` | 等待交接 |
| `Waiting Decision` | 等待拍板 |
| `In Review` | 审核中 |
| `Rework Required` | 需要返工 |
| `Ready for Delivery` | 可交付 |
| `Done` | 已完成 |
| `Archived` | 已归档 |
| `Cancelled` | 已取消 |

### 动作文案映射

| 系统 action | 默认用户可见文案 |
|---|---|
| `ready_task` | 标记为可开始 |
| `start_progress` | 开始执行 |
| `submit_handoff` | 提交交接 |
| `accept_handoff` | 接收交接 |
| `request_decision` | 请求拍板 |
| `resume_after_decision` | 按拍板结果继续 |
| `start_review` | 开始审核 |
| `reject_to_rework` | 驳回返工 |
| `restart_rework` | 开始返工 |
| `mark_ready_for_delivery` | 标记可交付 |
| `reopen_from_delivery` | 重新打开任务 |
| `complete_delivery` | 完成交付 |
| `archive_task` | 归档任务 |
| `cancel_task` | 取消任务 |

### 文案使用原则

- 页面标题、按钮文案、状态标签优先使用右侧用户文案
- 系统内部术语可以保留在代码、日志、接口层
- 默认不把内部 action key 直接显示给普通用户

## 第一阶段主要页面

### 1. 模板首页

#### 目标

让用户看到有哪些可直接启动的模板。

#### 主要内容

- 模板列表
- 模板摘要
- 推荐模板（软件公司模板）
- “从模板启动”入口

#### 主要跳转

- 进入模板详情页
- 启动 Workspace

---

### 2. 模板详情页

#### 目标

让用户理解模板包含哪些角色、workflow、任务样例和技能包。

#### 主要内容

- 模板说明
- 包含的角色
- 包含的 workflow
- 包含的任务样例
- 含有的默认技能包
- 启动按钮

#### 主要跳转

- 进入工作空间首页

---

### 3. 工作空间首页

#### 目标

让用户看到当前项目总体运行情况。

#### 主要内容

- 当前任务数
- 当前 Waiting Decision 数
- 当前 Rework 数
- 近期任务动态
- 快捷入口：任务列表 / 画布 / 审批页

#### 主要跳转

- 任务列表页
- Workflow 画布页
- 决策中心页

---

### 4. 任务列表页

#### 目标

集中查看与筛选任务。

#### 主要内容

- 任务列表
- `task_type` 筛选
- `status` 筛选
- `owner` 筛选
- 新建任务按钮

#### 主要跳转

- 任务详情页
- 新建任务页

---

### 5. 任务详情页

#### 目标

查看一个任务的完整业务信息与执行状态。

#### 主要内容

- 基础字段
- 验收标准
- 风险
- handoff 记录
- review / rework 记录
- 当前 workflow 摘要
- 当前可执行动作

#### 体验要求

- 默认先展示：当前状态、当前负责人、当前卡点、当前动作
- 记录默认展示摘要，不先把全部结构化原文砸给用户
- 用户应通过动作按钮推进，而不是靠理解内部状态名推进
- 状态标签和按钮文案应优先使用上面的 UI 文案映射

#### 主要跳转

- Workflow 画布页
- 决策中心页

---

### 6. Workflow 画布页

#### 目标

查看任务或 workflow 的可视化流转。

#### 主要内容

- 三栏布局
- workflow 画布
- 当前节点高亮
- 节点详情面板

#### 体验要求

- 画布主区优先帮助用户理解流程，不优先展示复杂内部参数
- 细节放到节点详情，不把主画布做成专家后台
- 节点详情里的动作应使用用户动作语言，而不是系统术语
- 状态节点和状态标签优先使用用户文案映射

#### 主要跳转

- 节点详情里的任务详情
- 决策页
- 角色与技能页（按需）

---

### 7. 决策中心页

#### 目标

集中处理 `Waiting Decision`。

#### 主要内容

- 待审批项列表
- 推荐方案
- 风险差异
- 同意 / 驳回 / 备注

#### 体验要求

- 用户应直接看到“为什么需要拍板”和“建议怎么选”
- 不要求用户先理解底层 decision record 结构
- 默认文案优先使用“拍板”而不是内部 decision 术语

#### 主要跳转

- 对应任务详情页
- 对应 workflow 画布页

---

### 8. 交付确认页

#### 目标

集中查看 `Ready for Delivery` 任务的交付摘要与最终确认信息。

#### 主要内容

- Delivery summary 摘要
- 当前交付对象
- 已知剩余风险
- 完成交付 / 重新打开任务入口
