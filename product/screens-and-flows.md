# product/screens-and-flows.md

## 目标
定义 AI Company Workflow 第一阶段的主要页面和页面流转，确保前端实现时：
- 知道有哪些页面
- 知道每个页面解决什么问题
- 知道页面之间怎么跳转
- 知道 PC 和手机的职责分工
- 知道如何把“系统内部复杂、用户使用简单”落实到页面结构

这份文档负责：
- 页面清单
- 页面目标
- 页面间跳转关系
- 页面层的体验约束

这份文档不负责：
- 画布页内部布局细节
- 节点 / 连线可视规则
- 画布详情侧栏的组件级规格

这些内容以 `product/canvas-ui-spec.md` 为准。

---

## 页面体验总原则
第一阶段的页面设计，要优先服务这几件事：
- 用户不用先理解内部对象模型，就能开始操作
- 用户一进入页面就能知道当前任务 / 当前卡点 / 下一步动作
- 默认页面只显示完成当前任务所需的信息
- 高复杂度信息进入二级页、详情区、抽屉或高级层

### 用户可见语言原则
页面默认应优先使用用户动作语言，例如：
- 从模板启动
- 开始执行
- 提交交接
- 请求拍板
- 驳回返工
- 标记可交付
- 完成交付

不应默认把这些内部术语直接作为主文案：
- transition
- payload
- guard
- workflow instance
- role assignment

### 信息暴露原则
默认页面优先展示：
- 现在是什么
- 卡在哪里
- 谁在负责
- 下一步做什么

默认不优先展示：
- 全量内部字段
- 全量记录原文
- 全量技能参数
- 复杂配置项

## UI 文案映射（Phase 1 默认口径）

### 状态文案映射
| 系统状态 | 默认用户可见文案 |
|---|---|
| `Draft` | 草稿 |
| `Ready` | 可开始 |
| `In Progress` | 进行中 |
| `Waiting Handoff` | 等待交接 |
| `Waiting Decision` | 等待拍板 |
| `In Review` | 审核中 |
| `Rework Required` | 需要返工 |
| `Ready for Delivery` | 可交付 |
| `Done` | 已完成 |
| `Archived` | 已归档 |
| `Cancelled` | 已取消 |

### 动作文案映射
| 系统 action | 默认用户可见文案 |
|---|---|
| `submit_ready` | 提交为可执行 |
| `start_progress` | 开始执行 |
| `submit_handoff` | 提交交接 |
| `accept_handoff` | 接收交接 |
| `request_decision` | 请求拍板 |
| `resume_after_decision` | 按拍板结果继续 |
| `start_review` | 开始审核 |
| `reject_to_rework` | 驳回返工 |
| `restart_rework` | 开始返工 |
| `mark_ready_for_delivery` | 标记可交付 |
| `reopen_from_delivery` | 重新打开任务 |
| `complete_delivery` | 完成交付 |
| `archive_task` | 归档任务 |
| `cancel_task` | 取消任务 |

### 文案使用原则
- 页面标题、按钮文案、状态标签优先使用右侧用户文案
- 系统内部术语可以保留在代码、日志、接口层
- 默认不把内部 action key 直接显示给普通用户

## 第一阶段主要页面

### 1. 模板首页
#### 目标
让用户看到有哪些可直接启动的模板。

#### 主要内容
- 模板列表
- 模板摘要
- 推荐模板（软件公司模板）
- “从模板启动”入口

#### 体验要求
- 新用户应一进来就知道“推荐先用哪个模板”
- 不要求先理解模板内部结构才能开始

#### 主要跳转
- 进入模板详情页
- 启动 Workspace

---

### 2. 模板详情页
#### 目标
让用户理解模板包含哪些角色、workflow、任务样例和技能包。

#### 主要内容
- 模板说明
- 包含的角色
- 包含的 workflow
- 包含的任务样例
- 包含的默认技能包
- 启动按钮

#### 体验要求
- 先解释“你将得到什么”，再展示更细的结构
- 复杂模板结构不应压过“从模板启动”主动作

#### 主要跳转
- 进入工作空间首页

---

### 3. 工作空间首页
#### 目标
让用户看到当前项目总体运行情况。

#### 主要内容
- 当前任务数
- 当前 Waiting Decision 数
- 当前 Rework 数
- 近期任务动态
- 快捷入口：任务列表 / 画布 / 审批页

#### 体验要求
- 用户一进入 workspace 首页，应立刻知道：
  - 现在系统有没有在跑
  - 卡在哪
  - 下一步先点什么
- 不应默认展示一大堆配置项或后台字段

#### 主要跳转
- 任务列表页
- Workflow 画布页
- 决策中心页

---

### 4. 任务列表页
#### 目标
集中查看与筛选任务。

#### 主要内容
- 任务列表
- task_type 筛选
- status 筛选
- owner 筛选
- 新建任务按钮

#### 体验要求
- 列表默认先让用户看到当前卡点和优先级
- 不要求用户理解全部状态机细节才能筛选任务

#### 主要跳转
- 任务详情页
- 新建任务页

---

### 5. 任务详情页
#### 目标
查看一个任务的完整业务信息与执行状态。

#### 主要内容
- 基础字段
- 验收标准
- 风险
- handoff 记录
- review / rework 记录
- 当前 workflow 摘要
- 当前可执行动作

#### 体验要求
- 默认先展示：当前状态、当前负责人、当前卡点、当前动作
- 记录默认展示摘要，不先把全部结构化原文砸给用户
- 用户应通过动作按钮推进，而不是靠理解内部状态名推进
- 状态标签和按钮文案应优先使用上面的 UI 文案映射

#### 主要跳转
- Workflow 画布页
- 决策中心页

---

### 6. Workflow 画布页
#### 目标
查看任务或 workflow 的可视化流转。

#### 主要内容
- 三栏布局
- workflow 画布
- 当前节点高亮
- 节点详情面板

#### 体验要求
- 画布主区优先帮助用户理解流程，不优先展示复杂内部参数
- 细节放到节点详情，不把主画布做成专家后台
- 节点详情里的动作应使用用户动作语言，而不是系统术语
- 状态节点和状态标签优先使用用户文案映射

#### 主要跳转
- 节点详情里的任务详情
- 决策页
- 角色与技能页（按需）

---

### 7. 决策中心页
#### 目标
集中处理 Waiting Decision。

#### 主要内容
- 待审批项列表
- 推荐方案
- 风险差异
- 同意 / 驳回 / 备注

#### 体验要求
- 用户应直接看到“为什么需要拍板”和“建议怎么选”
- 不要求用户先理解底层 decision record 结构
- 默认文案优先使用“拍板”而不是内部 decision 术语

#### 主要跳转
- 对应任务详情页
- 对应 workflow 画布页

---

### 8. 角色与技能页
#### 目标
查看角色和默认技能包。

#### 主要内容
- 角色列表
- 角色详情
- 默认技能包
- 技能启用状态

#### 体验要求
- 默认只展示角色会什么、适合什么、是否启用
- schema / binding / retry 等内部复杂参数不默认展开

---

## PC 端主流程
### 新用户启动
模板首页 -> 模板详情页 -> 启动 Workspace -> 工作空间首页

### 开始处理任务
工作空间首页 -> 任务列表页 -> 任务详情页 -> Workflow 画布页

### 审批拍板
工作空间首页 -> 决策中心页 -> 决策详情 -> 返回任务 / 画布

### 查看当前卡点
工作空间首页 -> Workflow 画布页 -> 当前节点详情

---

## 手机端主流程
手机端优先支持：
- 工作空间概览
- 当前任务状态
- Waiting Decision
- hotfix 摘要
- 轻量审批动作

### 手机端建议路径
工作空间首页 -> 决策中心页 -> 审批动作
工作空间首页 -> 当前任务卡片 -> 简化任务详情
工作空间首页 -> 当前流程缩略视图 -> 节点详情抽屉

### 手机端体验原则
- 手机端优先让用户快速判断与拍板
- 不要求手机端承载完整重配置能力
- 不要求用户在手机端理解复杂结构才能完成关键操作
- 手机上的状态与动作同样使用用户文案映射

---

## 页面间联动原则
- 任务详情与画布页要双向可跳
- 决策页与对应任务要双向可跳
- 工作空间首页始终作为总入口
- 手机端优先从“当前卡点”和“待拍板”进入，不要求完整重配置能力
- 页面间跳转要优先服务“继续推进下一步”，而不是优先展示更多配置

---

## 第一阶段不必急着做的页面
- 高复杂模板编排页
- 多维 BI 报表页
- 高复杂权限管理页
- 全功能移动端画布编辑页

这些页面不是不能做，而是当前 Phase 1 不应该让它们压过“模板启动 + 任务推进 + 拍板收口”的主链路体验。

---

## 一句话总结
第一阶段页面结构要服务一件事：

**用户能从模板启动，能看到任务进展，能在画布看懂流转，能在关键节点完成拍板，并且不需要先理解系统内部复杂度。**
