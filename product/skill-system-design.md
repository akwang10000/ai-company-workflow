# product/skill-system-design.md

## 目标
定义 AI Company Workflow 中的技能模块设计，明确：
- 为什么当前框架需要技能层
- 角色与技能如何分工
- 技能系统如何在工程内部保持严谨
- 界面如何避免暴露过多复杂参数
- 技能如何与 workflow、模板、画布协同

这份文档的目标不是把技能系统做成另一个复杂配置中心，而是把它设计成：
**内部强能力、外部简单可用、支持模板化复用的能力层。**

---

## 为什么当前框架适合引入技能模块
当前系统已经具备：
- 角色系统
- workflow 系统
- task schema
- 模板系统
- LLM 辅助创建方向

在这种结构下，如果只有角色，没有技能层，后续很容易出现：
- 角色定义越来越重
- 不同场景下角色能力难复用
- workflow 节点能力只能写死在角色文案里
- 模板扩展时难以灵活组合能力

因此当前框架适合引入技能模块，把：
- **角色** 负责身份、职责、边界
- **技能** 负责能力、方法、可复用逻辑

---

## 角色与技能的分工

### 角色回答的问题
- 你是谁
- 你负责什么
- 你不负责什么
- 你在 workflow 中处于哪一段

### 技能回答的问题
- 你会什么能力
- 你能处理什么类型的问题
- 你需要什么输入
- 你会产出什么输出
- 你适用于哪些角色 / workflow / 场景

### 原则
- 角色是职责容器
- 技能是能力模块
- workflow 是编排骨架

不要把三者混在一起。

---

## 技能系统的三层结构

## 第一层：内核层（工程内部）
这层可以复杂、严谨、可编排。

### 可包含的能力元数据
- `skill_id`
- `name`
- `description`
- `skill_type`
- `input_schema`
- `output_schema`
- `preconditions`
- `compatible_roles`
- `compatible_workflows`
- `side_effect_level`
- `risk_level`
- `tool_bindings`
- `timeout`
- `retry_policy`
- `fallback_policy`
- `version`
- `visibility_level`

### 原则
- 工程内部允许复杂，但必须结构化
- 复杂性应服务稳定性和可复用性，不应服务炫技
- 所有技能定义文件也必须统一使用 UTF-8 编码

---

## 第二层：编排层（产品逻辑层）
这层负责把技能接到角色、模板、workflow 上。

### 建议能力
- 角色默认挂载技能包
- workflow 节点声明需要哪些技能
- 模板可启用 / 禁用部分技能
- LLM 可根据用户描述推荐技能组合
- 技能能力可在不同角色间复用

### 示例
- PM Agent 默认技能包：
  - 需求澄清
  - 任务拆解
  - 验收标准生成
- Tech Lead Agent 默认技能包：
  - 技术评估
  - 风险识别
  - 模块拆分
- Dev Agent 默认技能包：
  - 实现修改
  - 自测摘要
  - 变更说明
- QA / Review Agent 默认技能包：
  - 验收检查
  - 回归检查
  - 返工单生成
- Ops / Release Agent 默认技能包：
  - 交付摘要
  - 归档整理
  - postmortem 初始化

---

## 第三层：界面层（用户可见层）
这层必须简单、克制、渐进暴露。

### 默认只建议暴露给用户的内容
- 技能名称
- 一句话作用说明
- 是否启用
- 适用角色 / 场景
- 是否为默认推荐技能
- 少量必要开关

### 不建议默认暴露给普通用户的内容
- input / output schema 细节
- timeout / retry / fallback 策略
- tool binding 细节
- prompt 片段
- 版本兼容与内部参数
- 高复杂度风险控制项

### 产品原则
- 默认简单
- 高级可展开
- 专家配置隐藏

---

## 用户界面的暴露策略

### 1. 普通模式
适合新用户，只看到：
- 角色
- workflow
- 技能卡片
- 开 / 关
- 简短说明
- 推荐组合

### 2. 进阶模式
适合愿意微调的用户，可看到：
- 输入要求
- 输出产物
- 触发条件
- 适用 workflow
- 风险等级

### 3. 专家 / 内部模式
只在必要时开放：
- schema
- tool binding
- timeout / retry / fallback
- 版本与内部配置

---

## 技能模块与模板系统的关系
第一阶段技能系统不建议让用户从零创建全部技能。
更合理的方式是：
- 先提供预制技能包
- 角色默认挂技能包
- 模板默认带技能组合
- 用户主要做启用 / 禁用 / 微调
- LLM 辅助生成技能草稿或推荐补充技能

### 最小启动模板中的技能建议
- 软件公司模板默认包含角色基础技能包
- feature / bugfix / hotfix workflow 默认绑定对应关键技能
- 任务样例可映射到建议技能组合

---

## 技能模块与工作流画布的关系
工作流画布不应把所有技能全部摊开成主节点，否则会把画布炸掉。

### 建议做法
#### 画布主视图展示
- 角色节点
- 审批节点
- 状态节点
- 流转关系

#### 节点详情中展示
- 当前角色默认技能
- 当前节点启用技能
- 输入 / 输出
- 风险 / 限制
- 当前执行记录

### 原则
- 画布先服务理解与推进
- 技能信息做成节点详情层，而不是主画布噪音层

---

## 与 LLM 辅助创建的关系
技能系统应支持：
- 用户描述业务或团队需求
- LLM 生成角色草稿
- LLM 推荐技能组合
- 用户微调后保存为模板

### 约束
- LLM 负责起草和推荐
- 用户负责确认
- 复杂参数不默认暴露
- 技能一旦进入模板，应可复用和版本化

---

## 当前阶段的设计结论
对于当前 AI Company Workflow 框架：
- 引入技能模块是合理的
- 工程内部可以复杂严谨
- 用户界面必须少暴露参数
- 应以模板化默认技能包为主，而不是空白自定义为主
- 技能要作为角色和 workflow 的能力补层，而不是抢掉角色和 workflow 本身的职责

---

## 不该做成什么样
应避免把技能系统做成：
- 另一个复杂配置中心
- 比 workflow 还复杂的参数页面
- 需要普通用户理解 schema / retry / binding 才能使用的系统
- 把角色、技能、workflow 三层概念搅在一起的系统

---

## 一句话原则
技能模块适合当前框架，但应遵循：
**内核复杂、界面克制、模板优先、渐进暴露、普通用户可用。**
