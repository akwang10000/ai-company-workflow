# product/product-implementation.md

## 目标
定义 AI Company Workflow 第一阶段（研发团队版）的产品实施约束，确保后续原型、后端、前端和交互设计沿同一条实现路径推进，而不是各写各的。

这份文档回答的是：
- 第一版产品要做成什么形态
- 技术实现优先怎么选
- 前后端分别承担什么职责
- 哪些实现约束现在就要提前定下来

---

## 第一阶段实施范围
当前第一阶段只聚焦 **研发团队版**，目标不是一次做成全行业平台，而是先把“第一个软件公司模板”做成可运行、可演示、可复用的产品骨架。

第一阶段重点包括：
- 角色模板管理
- 工作流模板管理
- 任务单管理
- 执行状态流转
- 决策门 / 审批点
- 工作流画布展示与操作
- 基础任务样例与最小启动模板
- 技能模块的模板化挂载与启停
- 当前项目自身的实施蓝图文档

不在第一阶段强求：
- 全行业模板市场
- 高复杂度权限体系
- 大规模自动化编排引擎
- 深度 BI / 报表系统
- 多租户企业级治理全量能力

---

## 技术选型约束

### 后端
第一阶段后端优先采用：
- **Java + Spring Boot**

### 选择原因
- 适合承载任务流转、状态机、审批流、模板管理等强业务建模需求
- 生态成熟，便于后续扩展权限、审计、消息通知、持久化和集成能力
- 对“流程系统 / 企业应用 / 工作台”类产品长期演进更稳

### 后端职责建议
后端优先承担：
- 角色模板、workflow 模板、task 模板的管理
- 技能模块及角色/模板绑定关系管理
- 任务状态流转
- 决策门与审批记录
- 执行日志与审计记录
- 用户配置与模板实例化
- 画布节点/边的结构化数据存储
- 对 LLM 辅助创建角色 / workflow / 技能组合的服务编排接口

---

## 前端形态约束

### 前端目标
前端第一阶段必须优先支持：
- **PC + 手机双端可用**

### 原则
不是“PC 做完整、手机凑合能看”，而是要从产品设计阶段就考虑：
- PC 适合搭建、查看全局、编辑模板、操作画布
- 手机适合查看进度、接收提醒、审批拍板、做轻量编辑

### 前端应支持的核心场景
#### PC 端优先场景
- 新建/编辑角色模板
- 新建/编辑工作流模板
- 查看完整工作流画布
- 配置节点、角色、交接、审批门
- 配置默认技能包与模板能力组合
- 查看任务全生命周期

#### 手机端优先场景
- 查看当前任务状态
- 查看谁卡住了、下一步是谁
- 接收审批提醒 / 决策提醒
- 快速同意 / 驳回 / 备注
- 查看 hotfix / bugfix 的进展摘要
- 查看当前节点启用的关键技能和执行摘要

### 前端设计要求
- 不允许关键流程只能在单一端完成
- PC 与手机的信息架构应共享同一套核心模型
- 手机端可适当弱化复杂编辑，但不能缺失关键查看和拍板能力
- 默认界面要少参数暴露，复杂配置折叠到高级层

## 工作流画布实施要求

### 目标
工作流画布不是装饰层，而是产品的核心操作界面之一。

### 第一阶段画布至少应支持
- 节点展示：角色节点、任务节点、审批节点、状态节点
- 连线展示：标准流转、返工流转、升级决策流转
- 当前状态高亮
- 当前负责人可见
- 点击节点查看详细信息
- 支持从模板实例化出工作流

### 体验要求
- 要有足够沉浸感，让用户一眼看懂“任务现在走到哪了”
- 要优先考虑易懂和易操作，不要做成复杂 BPMN 专家工具
- 默认模板应能直接落在画布里，而不是只存在文档说明中
- 技能信息优先放在节点详情层展示，不把主画布做炸

## 模板驱动实施要求
第一阶段必须坚持：
- 模板优先，而不是空白优先
- 新用户优先从“第一个软件公司模板”启动
- 角色、workflow、task 都要有预制模板
- 技能优先以默认技能包形式随角色和模板一起交付

### 最小启动模板建议包含
- PM Agent
- Tech Lead Agent
- Dev Agent
- QA / Review Agent
- Ops / Release Agent
- feature workflow
- bugfix workflow
- hotfix workflow
- 3 类任务样例
- 一组角色基础技能包

---

## LLM 辅助创建要求
第一阶段应预留并逐步支持：
- 用户通过自然语言描述团队或业务场景
- LLM 生成角色模板初稿
- LLM 生成 workflow 草稿
- LLM 推荐技能组合与模板骨架
- 用户微调后保存为正式模板

### 原则
- LLM 负责起草，不直接替用户拍板
- 用户可以微调角色职责、输入输出、审批边界
- 模板一旦确认，应可固化并复用
- 复杂参数默认不暴露给普通用户

## 第一阶段完成后的理想效果
如果第一阶段实现到位，用户应能：
- 直接选择一个软件公司最小启动模板
- 在 PC 上快速查看和调整角色 / workflow
- 在手机上随时查看进度并拍板
- 用对话方式辅助生成新角色或新 workflow
- 用统一任务单驱动 feature / bugfix / hotfix 流程
- 使用默认技能包而不是手搓复杂能力配置
- 在问题结束后完成归档与复盘

---

## 当前实施优先级建议
1. 固化后端模型：角色模板 / workflow 模板 / 技能模块 / 任务单 / 状态流转
2. 固化前端信息架构：PC + 手机共享同一核心模型
3. 做最小画布原型：先能看、能点、能追踪
4. 做模板启动流程：先让用户不从空白开始
5. 再逐步补 LLM 辅助创建能力

更细的实施顺序继续参考：
- `product/implementation-roadmap.md`
- `product/implementation-phases.md`

## 一句话原则
第一阶段不是做“最强配置系统”，而是做一套：
**后端稳、前端双端可用、模板可直接启动、画布好理解、技能可控且不炸界面、普通用户也能上手，并且 Codex 能按阶段持续施工的 AI 团队工作流产品。**
